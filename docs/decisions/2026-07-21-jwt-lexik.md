# Décision — Authentification JWT via lexik/jwt-authentication-bundle

Date : 2026-07-21  
Contexte : login local sur `CoreCredential`

## Choix

Utiliser **lexik/jwt-authentication-bundle** (standard Symfony pour JWT) plutôt
qu'une implémentation maison. Bundle mature, intégration Security native
(`json_login` + authenticator `jwt`).

## Structure provider / UserInterface

Symfony Security exige un `UserInterface` (+ `PasswordAuthenticatedUserInterface`
pour le login mot de passe). L'agrégat Domain `CoreCredential` **ne doit pas**
implémenter ces interfaces (ADR-002 / pureté Domain).

Solution retenue :

1. `SecurityUser` (Infrastructure) — adapter qui enveloppe email, hash, ids
2. `CoreCredentialUserProvider` — `PartyAccountRepository::findByEmail()` puis
   résolution du `CoreCredential` local actif associé
3. Vérification mot de passe via le hasher Symfony configuré sur `SecurityUser`
   (même algo `auto` que `SymfonyPasswordHasher` Domain)

Pour le login local, l'identifiant n'est **pas** `provider_user_id` (NULL) mais
l'**email** du `party_account`.

## Payload JWT (ADR-018)

Claims exposés au client : `username` (email, Lexik) + `public_id` (enrichissement).
**Pas** d'`account_id` BIGINT — un JWT est signé, pas chiffré ; l'id interne
reste côté serveur dans `SecurityUser` uniquement.

## Anti-énumération

Échec email inconnu, mot de passe incorrect **et** compte désactivé → **même 401**
Lexik (`Invalid credentials.`). `hide_user_not_found` Symfony (défaut) + réponse
failure Lexik unique. Ne jamais préciser lequel des cas est en cause.

## Périmètre

- Firewall `login` : `POST /api/v1/auth/login`
- Firewall `api` : `/api/v1/*` JWT stateless
- Hors API (`main`) : `security: false` (futur back-office HTML hors scope)
