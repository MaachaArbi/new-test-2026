# TASK — Configuration CORS pour consommation front

## Contexte
Aucun front n'existe encore, mais il faut préparer l'API à être appelée
depuis un navigateur (futur front jetable puis le vrai). Sans CORS, le
navigateur bloque toute requête cross-origin avant même qu'elle atteigne
Symfony — invisible dans les logs serveur, source de confusion si découvert
tard.

## Choix technique
nelmio/cors-bundle (standard Symfony, gère preflight OPTIONS automatiquement).

## Configuration
config/packages/nelmio_cors.yaml :
- allow_origin : liste explicite en variable d'environnement (CORS_ALLOW_ORIGIN
  dans .env), PAS de wildcard '*' — même en dev, prendre l'habitude de lister
  les origines explicitement (ex: http://localhost:5173 pour un futur Vite).
  Documenter dans .env.example avec une valeur d'exemple claire.
- allow_headers : au minimum Content-Type, Authorization (obligatoire pour
  le Bearer JWT)
- allow_methods : GET, POST, PUT, PATCH, DELETE, OPTIONS
- expose_headers : X-Request-Id (pour que le front puisse le lire en cas
  d'erreur, cohérent avec notre observabilité)
- paths : appliquer uniquement sur ^/api/ (pas sur d'éventuelles futures
  routes non-API)

## Test
Un test d'intégration qui vérifie qu'une requête OPTIONS (preflight) sur un
endpoint existant avec un header Origin autorisé reçoit bien les headers
Access-Control-* attendus, et qu'une origine NON listée dans
CORS_ALLOW_ORIGIN ne les reçoit pas (vérifier le comportement de rejet,
pas juste le succès).

## Documentation

Relance phpstan/deptrac/phpcpd/phpunit. Colle le contenu intégral des fichiers

autorisée (celui-là en particulier, pas de résumé).
