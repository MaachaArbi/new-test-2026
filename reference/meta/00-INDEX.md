## Principe directeur pour tout nouveau module (ajouté 16/07)

**Pas de "legacy amélioré".** Pour Party, Booking et Règlements, une bonne partie de la structure legacy (`ost_amicale`, `ost_client`, réservations par service) a servi de matière première à challenger. Pour tous les modules restants, la consigne explicite de l'utilisateur est différente : le legacy est **faible** sur ces sujets (manque de maîtrise métier, incohérences), et ne doit servir que de liste de fonctionnalités à confronter — jamais de gabarit structurel à reproduire. Chaque nouveau module doit partir d'une conception repensée à partir du besoin métier réel, pas d'un mapping table-à-table de l'existant.

Exception explicite : **Booking, Contracting et les intégrations API/providers** sont identifiés par l'utilisateur comme la partie la plus solide du système legacy — sur ces sujets, challenger l'existant reste utile et le legacy fait davantage autorité que sur le reste.

**Principe transverse acté le 18/07** : la base de données porte les invariants structurels (contraintes d'intégrité, plafonds, unicité — protection contre tout écrivain, y compris futurs scripts/services). La logique métier réelle (calculs, orchestration, règles qui évoluent) reste en couche Domain PHP/Symfony (ADR-002), jamais en procédure stockée. Exception documentée : `log_audit_trigger()` (capture mécanique avant/après, aucune règle métier — ADR-006).

**Règle transverse acquise en session Pricing (19/07)** : avant de construire une table pour un concept qui *semble* générique (regroupement, catégorie, dimension transverse...), vérifier explicitement s'il n'est pas déjà anticipé dans un module de couche plus basse (Party, `ref_static`) — chercher dans `sujets-reportes.md` et les `modele-conceptuel-*.md` avant de construire, pas après.

**Discipline de fin de session (rappel renforcé le 20/07, suite à deux incidents de désynchronisation)** : toute session qui régénère `00-INDEX.md`/`sujets-reportes.md` doit repartir des fichiers **réellement présents dans le Project au moment de la clôture**, pas d'une copie locale conservée depuis le début de la session — surtout si la session a duré longtemps ou a été relancée après une pause. En cas de doute, le chat pilote vérifie et fusionne manuellement plutôt que de laisser écraser un fichier maître par une version obsolète.

## Stratégie de migration — legacy en API Gateway temporaire (ajoutée 18/07)

Décision structurante de l'utilisateur, distincte de l'ordre de conception BDD mais qui l'influence directement : la partie la plus complexe et la plus risquée du legacy (**Contracting hôtelier** — saisie des tarifs d'achat et calculs associés, y compris les micro-marges de contrat par arrangement/politique enfant/réduction chambre, `sujets-reportes.md` §52) ne sera **pas migrée en premier**. Le système legacy continuera à servir de passerelle API (« API gateway ») pendant plusieurs mois pendant que tout le reste bascule vers la nouvelle application. Raison explicite : risque direct sur de l'argent et sur des engagements client réels.

**Sur 2-3 mois, les deux systèmes tournent en parallèle** (décision actée 19/07) : legacy reste seul maître de la vérité financière (règlements/factures saisis manuellement en double par l'équipe), import automatique limité aux données de référence/mapping (produits, clients, users — désormais tous figés).

**Précision actée (18/07)** : les **marges de vente** (Pricing) sont explicitement **découplées** du contracting hôtelier — confirmé sur écrans legacy réels. Pricing garde sa place normale dans l'ordre, n'est pas repoussé.

## Note de resync (22/07/2026) — 4ᵉ incident de désynchronisation, résolu

**Symptôme** : après le ré-upload des fichiers censés porter l'amendement ADR-018 v2.1, le chat pilote a rejoué la chaîne depuis les fichiers **réellement présents** et obtenu 290 tables (et non 292), avec trois écarts par rapport à ce que déclarait cet INDEX.

**Cause racine** : suppression-sans-remplacement lors du dernier upload de l'amendement ADR-018 (pas un défaut de génération), **plus** trois `diff` jamais réellement bakés dans leurs schémas de base alors que l'INDEX les décrivait comme appliqués :
- `diff-booking-log-generalization.diff` non appliqué → `booking_log` coexistait avec `log_activity`/`log_audit` (vrai doublon fonctionnel, et +1 table). C'est ce doublon qui gonflait le compte à 292.
- `diff-pricing-payment-modality.sql` (pseudo-diff à en-tête de hunk non standard, non exécutable tel quel) jamais intégré → `pricing_payment_party_role`, `pricing_payment_modality_detail`, nature `payment_modality` absentes.
- `ref-common-hebergement-extension.diff` non appliqué → `schema-ref-common.sql` restait en V1.1 (pas de `native_name`/`alpha3`/`oct_code`).

**Correctif appliqué le 22/07** : les 3 diffs bakés dans leurs bases (édition directe des 3 schémas figés, sur feu vert utilisateur), l'amendement ADR-018 v2.1 vérifié intact (les 4 tables partitionnées `booking`/`core_session`/`core_auth_attempt`/`provider_call_log` toujours en `GENERATED BY DEFAULT`). Chaîne rejouée : **16/16, 0 erreur, 291 tables**, `booking_log` disparu, `payment_modality` présente, ref-common en V1.2. Les 3 fichiers `.diff`/`.sql` d'origine restent comme trace historique (statut « baké », comme les 6 autres diffs du projet).

**Garde-fou renforcé** : un diff n'est « appliqué » que si son contenu est **vérifié présent dans le schéma de base** (grep de la table/colonne cible), jamais sur la seule foi d'une mention dans l'INDEX. Tout `diff-*.sql` doit être un vrai script exécutable OU être baké — pas un pseudo-diff flottant.

## Décisions structurantes à retenir

- **Tiering** : Critique / Standard / Référentiel — traitement proportionné.
- **Multilingue** : contenu uniquement (libellés), jamais les données métier. 3 langues (EN pivot, FR, AR).
- **Nommage** : préfixe par module (`party_`, `core_`, `ref_`, `booking_`, `settlement_`, `log_` (transverse), `cash_`, `sales_point_`, `invoicing_`, `product_`, `pricing_` — futur `provider_`) dans un schéma PostgreSQL unique. Incohérence connue et non corrigée : `settlement_`/`sales_point_` restent en français (`sujets-reportes.md` §39).
- **Identifiants** : BIGINT identity + `public_id` UUID sur toutes les tables (ADR-018).
- **Money** : BIGINT minor units (`ref_currency.minor_unit`), sauf `pricing_*` en `NUMERIC(12,4)` (nature différente : taux/marge, pas un montant transactionnel définitif).
- **Isolation** : 1 serveur = 1 client, jamais de `tenant_id` (ADR-004).
- **Partitionnement** : `booking_`, `core_session`, `core_auth_attempt` partitionnées par date. `pg_partman` à automatiser avant prod (jamais fait à ce jour).
- **Faits immuables** : Booking constate des faits datés ; Règlements projette ces faits dans son grand livre append-only. Règlements lit Booking, n'y écrit jamais.
- **Grand livre Règlements** : append-only garanti par trigger. Snapshot `settlement_balance`. Toute correction = contre-passation datée.
- **Cash Management** : même ADN que Règlements. Extension de référentiel par table compagnon 1-1 plutôt que réouverture du module propriétaire.
- **Référentiel Hébergement & Géographie** : miroir local (PULL uniquement) d'OctaSoft Static Data (produit séparé, hors périmètre) + ajout local selon l'entité (`oct_code NULLABLE` vs `NOT NULL`). `content_provider` = entité fondatrice pour Provider Integration.
- **Tables de référence, pas d'ENUM** : tout ce qui est catégoriel et extensible est une table — y compris pour de petits ensembles fixes (ex: `pricing_payment_party_role`, `log_entity_type`).
- **EAV/tables génériques rejetées par principe**, y compris pour la configuration applicative (`config_application_setting` = singleton à colonnes explicites, pas clé/valeur).
- **Log transverse** (ADR-006, enfin construite 20/07) : `log_activity` (métier, alimentée par l'Application) vs `log_audit` (technique ligne-par-ligne, alimentée par un trigger générique réutilisable — seule exception documentée à ADR-002). `entity_id` toujours en FK applicative. Piège connu : `TG_TABLE_NAME` retourne le nom de partition physique, pas le nom logique.
- **Pricing** : garde-fou structurel anti-mélange entre natures de règle (`margin`/`commission`/`payment_modality`) par FK composite `(rule_id, rule_nature_code)` — pattern à reconduire pour toute future nature. Résolution de conflit entre règles concurrentes : la plus récemment créée l'emporte, aucune notion de spécificité, entièrement en Domain (jamais en base).
- **RBAC** (ADR-017) : opt-in inversé, `core_permission` ≠ `party_role` (deux notions distinctes, jamais confondues). RBAC ne fait pas de filtrage de données (porté par `party_account_function`).

## Méthode de conception (à reconduire sur le dernier module restant)

1. Proposer une structure précise à partir du contexte disponible.
2. Tester chaque hypothèse sur PostgreSQL réel (sandbox container) — **chaîne complète, pas seulement le nouveau module**, pour détecter toute régression.
3. Confronter à des données réelles exportées — les cas réels doivent pouvoir casser une hypothèse.
4. Documenter chaque revirement avec sa justification dans le modèle conceptuel.
5. **Avant de régénérer `00-INDEX.md`/`sujets-reportes.md` en fin de session, relire le fichier réellement présent dans le Project au moment de la clôture** (pas une copie locale du début de session) — deux incidents de désynchronisation déjà rencontrés et corrigés (voir `sujets-reportes.md` §19 et §55).
